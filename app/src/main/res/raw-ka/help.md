## SSH Remote-ის შესახებ

ეს ქართული თარგმანი შეიქმნა GitHub Copilot-ის დახმარებით და შესაძლოა შეიცავდეს თარგმანის შეცდომებს.

SSH Remote არის უფასო და ღია კოდის მქონე აპი, რომელიც SSH-ის გამოყენებით კომპიუტერების დისტანციურად მართვის საშუალებას გაძლევთ.

შეგიძლიათ სრულად მოარგოთ შესასრულებელი ბრძანებები და ასევე ხელმისაწვდომია პრესეტები გავრცელებული კონფიგურაციებისთვის.

მე ამ აპს ვიყენებ ჩემი HTPC სისტემის სამართავად, რომელიც Raspberry Pi OS-ზე მუშაობს. HTPC-ის მართვა ის ძირითადი სცენარია, რომლისთვისაც ეს აპი ყველაზე მეტად არის ოპტიმიზებული.

ეს აპი ტერმინალის ემულატორი არ არის, მაგრამ გადაუდებელ სიტუაციაში `apt-get install`-ის გაშვების საშუალებას მოგცემთ.

## დაწყება

თუ კავშირისათვის SSH გასაღების გამოყენება გსურთ, ჯერ გახსენით აპის პარამეტრები და დააიმპორტეთ ან დააგენერირეთ გასაღები.

დაამატეთ ახალი ჰოსტი ქვედა მარჯვენა კუთხეში მდებარე `+` ღილაკზე შეხებით. შეიყვანეთ კავშირის დეტალები და შეინახეთ.

პირველი დაკავშირებისას მოგეთხოვებათ პრესეტის არჩევა. ეს არჩევანი დააკონფიგურირებს პულტის ღილაკებს ისე, რომ ისინი სხვადასხვა ტიპის კომპიუტერზე კარგად იმუშაოს. ქვემოთ იხილეთ ხელმისაწვდომი პრესეტების აღწერა. თუ გსურთ, შეგიძლიათ დაიწყოთ სრულიად ცარიელი კონფიგურაციით `პრესეტის გარეშე` არჩევით.

თუ არ იცით, თქვენს Linux კომპიუტერზე X11 მუშაობს თუ Wayland, ტერმინალში გაუშვით ეს ბრძანება:

```shell
echo $XDG_SESSION_TYPE
```

შედეგად უნდა მიიღოთ `x11` ან `wayland`. ეს ბრძანება უნდა გაუშვათ უშუალოდ desktop გარემოში.

## პულტი

დაკავშირების შემდეგ შეგიძლიათ გამოიყენოთ პულტის ინტერფეისი ბრძანებების გასაგზავნად. სხვადასხვა შეყვანის მეთოდებზე გადასასვლელად შეცვალეთ ჩანართები.

ყოველი ღილაკის დაჭერა ჰოსტზე ერთ ბრძანებას ასრულებს. ასეთ მარტივ მოქმედებაზე, როგორიც ღილაკის დაჭერაა, ეს საკმაოდ დიდ ზედნადებს ქმნის და ჩვეულებრივ კლავიატურასთან შედარებით შეიძლება საკმაოდ მაღალი დაყოვნება შეამჩნიოთ. იმედი მაქვს, მომავალ ვერსიებში ამას გავაუმჯობესებ.

მენიუს გამოყენებით შედით რედაქტირების რეჟიმში. ამჟამად განლაგებისა და ღილაკების ხატულების რედაქტირება შეუძლებელია. იმედი მაქვს, მომავალ ვერსიაში ეს შესაძლებელი გახდება.

## პრესეტები

თქვენ უნდა დააყენოთ ის ინსტრუმენტი, რომელიც თქვენს desktop გარემოს სჭირდება.

მე გირჩევთ `ydotool`-ს, რადგან ჩემი ტესტების მიხედვით მას საუკეთესო წარმადობა აქვს და მუშაობს როგორც X11-ზე, ისე Wayland-ზე.

### ydotool

`ydotool` უნდა მუშაობდეს ნებისმიერი window manager-ით, მაგრამ მას სჭირდება ფონური სერვისი. თუ თქვენს დისტრიბუციას systemd-ის მომხმარებლის სერვისი მოაქვს, გაუშვით ის ასე:

```shell
systemctl start --user ydotool
```

სერვისის შესვლისას ავტომატურად გასაშვებად ჩასართავად გაუშვით:

```shell
systemctl enable --user ydotool
```

გთხოვთ, დარწმუნდეთ, რომ აყენებთ `ydotool`-ის საკმარისად ახალ ვერსიას. Ubuntu-ის 26.04-მდე ვერსიებში ხელმისაწვდომი ვერსიები ზედმეტად ძველია. გამოსავლისთვის იხილეთ დისკუსიის თემა.

გთხოვთ, `ydotool`-ის შესახებ იმსჯელოთ ამ დისკუსიის თემაში: <https://github.com/stefansundin/SSHRemote/discussions/5>

### xdotool

`xdotool` განკუთვნილია X11-ზე მომუშავე კომპიუტერებისთვის. ისტორიულად Linux კომპიუტერების უმეტესობა X11-ს იყენებდა, თუმცა Wayland სულ უფრო პოპულარული ხდება.

X11-ის ერთი თავისებურებაა, რომ შეიძლება მოგიწიოთ X სერვერზე წვდომის დაშვება. თუ იღებთ `Authorization required` შეცდომებს, სავარაუდოდ პრობლემა სწორედ ესაა. ამ საკითხის მოსაგვარებლად რამდენიმე ვარიანტი არსებობს; აი, ორი ვარიანტი, რომლებმაც ჩემთან იმუშავა:

თუ `xauth list` ჩანაწერებს არ აჩვენებს, სცადეთ `.Xauthority` ფაილის გენერირება:

```shell
xauth generate :0 . trusted
```

თუ ამან არ იმუშავა, მაშინ სცადეთ წვდომის მინიჭება `xhost`-ით:

```shell
xhost +local:$USER
```

`xhost` ბრძანების გაშვება ყოველი ჩატვირთვის შემდეგ დაგჭირდებათ. ამის ავტომატიზაცია შეგიძლიათ bash სკრიპტის შექმნით და შესვლისას ავტომატურად გასაშვებად დაყენებით.

გთხოვთ, `xdotool`-ის შესახებ იმსჯელოთ ამ დისკუსიის თემაში: <https://github.com/stefansundin/SSHRemote/discussions/3>

### wtype

`wtype` არის რაღაცით `xdotool`-ის მსგავსი, მაგრამ Wayland-ისთვის.

ჩვეულებრივ მას თაგვის მართვის მხარდაჭერა არ აქვს, თუმცა მე შევქმენი მოდიფიცირებული ვერსია, რომელიც თაგვის მხარდაჭერას ამატებს. თუ თაგვის მხარდაჭერა გჭირდებათ, გთხოვთ დააყენოთ ის: <https://github.com/stefansundin/wtype>

თუ იღებთ შეცდომას `Compositor does not support the virtual keyboard protocol`, გირჩევთ სცადოთ სხვა ინსტრუმენტი, მაგალითად `ydotool`.

გთხოვთ, `wtype`-ის შესახებ იმსჯელოთ ამ დისკუსიის თემაში: <https://github.com/stefansundin/SSHRemote/discussions/4>

### dotool

`dotool` უნდა მუშაობდეს ნებისმიერი window manager-ით, `ydotool`-ის მსგავსად. ჩემი შეზღუდული ტესტების მიხედვით, ის `ydotool`-ზე ბევრად ნელი იყო.

გთხოვთ, `dotool`-ის შესახებ იმსჯელოთ ამ დისკუსიის თემაში: <https://github.com/stefansundin/SSHRemote/discussions/6>

### cec-client

ეს პრესეტი ექსპერიმენტულია, რადგან ჩემს აპარატურაზე მისი შემოწმება ვერ მოვახერხე. უკუკავშირი მისასალმებელია.

გთხოვთ, `cec-client`-ის შესახებ იმსჯელოთ ამ დისკუსიის თემაში: <https://github.com/stefansundin/SSHRemote/discussions/7>

### macOS

არსებობს პრესეტი macOS-ზე VLC-ის სამართავად AppleScript ბრძანებების გამოყენებით. მე არ ვიცი რაიმე ინსტრუმენტი, რომელიც კლავიატურის ან თაგვის მოვლენების გაგზავნას უჭერს მხარს.

გთხოვთ, macOS-ის შესახებ იმსჯელოთ ამ დისკუსიის თემაში: <https://github.com/stefansundin/SSHRemote/discussions/8>

### Android

თუ თქვენს Android მოწყობილობას მოჰყვება SSH სერვერი, შესაძლოა შეძლოთ მასთან დაკავშირება და შეყვანის მოვლენების გაგზავნა. ეს უფრო მოსალოდნელია, თუ თქვენი მოწყობილობა იყენებს მორგებულ ROM-ს, მაგალითად Raspberry Pi-ზე KonstaKANG-ს.

ჯერ ვერ მოვახერხე, როგორ გავამუშაო თაგვის მხარდაჭერა.

გთხოვთ, Android-ის შესახებ იმსჯელოთ ამ დისკუსიის თემაში: <https://github.com/stefansundin/SSHRemote/discussions/9>

## ხმის ჭკვიანი კონტროლი

პულტის რედაქტირებისას მენიუში შეგიძლიათ იპოვოთ ხმის "ჭკვიანი" კონტროლის პარამეტრები. ეს საშუალებას გაძლევთ აპში ნახოთ კომპიუტერის მიმდინარე ხმა და სლაიდერით სწრაფად დააყენოთ ახალი მნიშვნელობა. ასევე შეგიძლიათ თქვენი მოწყობილობის აპარატურული ღილაკებით სწრაფად გააგზავნოთ ხმის აწევის/დაწევის ბრძანებები.

მიმდინარე ხმის წაკითხვა და სლაიდერით ახალი ხმის დაყენება ამჟამად მყარად არის მიბმული `pactl`-ის გამოყენებაზე.

პაკეტს, რომელიც `pactl`-ს შეიცავს, ჩვეულებრივ ჰქვია `pulseaudio-utils` ან `libpulse`.

## SSH გასაღებები

აპის პარამეტრებში შეგიძლიათ SSH გასაღებების იმპორტი ან გენერირება. SSH გასაღებით დაკავშირება პაროლის გამოყენებაზე უფრო უსაფრთხოა.

არსებული SSH გასაღების კომპიუტერიდან იმპორტის ყველაზე მარტივი გზა QR კოდის სკანირებაა. QR კოდის გამოსახულების შესაქმნელად შეგიძლიათ გამოიყენოთ `qrencode` პროგრამა. QR კოდის გენერირებისთვის გაუშვით მსგავსი ბრძანება:

```shell
# გადადით თქვენი SSH გასაღებების საქაღალდეში:
cd ~/.ssh

# აჩვენეთ QR კოდი ტერმინალში:
qrencode -r id_ed25519 -t ansiutf8

# ალტერნატიულად, შექმენით გამოსახულების ფაილი:
qrencode -r id_ed25519 -o qr.png

# 4096-ბიტიანი RSA გასაღებები QR კოდში ვერ ეტევა. gzip-ის გამოყენებით ერთ-ერთს ზღვარზე ჩაატევთ:
cat id_rsa_4096 | gzip | base64 | qrencode -o qr.png
```

ჰოსტზე საჯარო SSH გასაღებების გადატანა შეგიძლიათ მენიუში არსებული `საჯარო გასაღების ატვირთვა` ფუნქციით. ეს არჩეულ SSH გასაღებს დაამატებს `~/.ssh/authorized_keys` ფაილის ბოლოში. ასე მარტივად შეგიძლიათ პაროლით შესვლიდან SSH გასაღებით შესვლაზე გადართვა.

შეგიძლიათ დააიმპორტოთ და გამოიყენოთ დაშიფრული SSH გასაღებები, თუმცა მათი გენერირება აპში ამჟამად შეუძლებელია.

## უსაფრთხოება

აპიდან შეუძლებელია SSH გასაღებების პირადი ნაწილის ან შენახული პაროლების ექსპორტი ან ამოღება. ეს მონაცემები დაშიფრულია 256-ბიტიანი AES-ით, ხოლო დაშიფვრის გასაღები ინახება Android Keystore-ში. დაშიფრული მონაცემები Android-ის სარეზერვო ასლებიდან გამორიცხულია.

ამ აპში არ არის ავარიების რეპორტინგის პროგრამული უზრუნველყოფა. არ არის ტელემეტრია. არ არის რეკლამები. არ არის ქსელური მოთხოვნები, გარდა SSH კავშირისა.

ამ აპის უსაფრთხოება აუდიტირებული არ არის. თუ Android-ის ან SSH-ის უსაფრთხოებაში გამოცდილი ხართ, გთხოვთ გაეცნოთ საწყის კოდს და თქვენი აღმოჩენები მოგვწეროთ GitHub-ის ამ issue-ში:
