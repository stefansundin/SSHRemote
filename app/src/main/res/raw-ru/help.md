## О SSH Remote

Этот перевод выполнен с помощью GitHub Copilot; в нём могут встречаться ошибки перевода.

SSH Remote — это бесплатное приложение с открытым исходным кодом, которое позволяет удалённо управлять компьютерами по SSH.

Вы можете полностью настроить выполняемые команды, а для распространённых сценариев доступны предустановки.

Я использую это приложение для управления своим HTPC, работающим на Raspberry Pi OS. Управление HTPC — это основной сценарий, под который оптимизировано приложение.

Это приложение не является эмулятором терминала, но в экстренной ситуации позволит вам выполнить `apt-get install`.

## Начало работы

Если вы хотите подключаться с помощью SSH-ключа, сначала откройте настройки приложения и импортируйте или сгенерируйте ключ.

Добавьте новый хост, нажав кнопку `+` в правом нижнем углу. Введите данные для подключения и сохраните их.

При первом подключении вам будет предложено выбрать предустановку. Этот выбор настраивает кнопки пульта так, чтобы они хорошо работали с разными типами компьютеров. Ниже приведено описание доступных предустановок. При желании можно начать без какой-либо конфигурации, выбрав `Без предустановки`.

Если вы не знаете, использует ли ваш Linux-компьютер X11 или Wayland, выполните в терминале следующую команду:

```shell
echo $XDG_SESSION_TYPE
```

В результате должно быть выведено `x11` или `wayland`. Эту команду необходимо выполнять внутри графической среды рабочего стола.

## Пульт дистанционного управления

После подключения вы можете использовать интерфейс пульта для отправки команд. Переключайте вкладки, чтобы получить доступ к различным способам ввода.

Каждое нажатие кнопки приводит к выполнению команды на хосте. Для такой простой операции, как нажатие клавиши, это создаёт заметные накладные расходы, поэтому вы можете столкнуться с довольно высокой задержкой по сравнению с обычной клавиатурой. Я надеюсь улучшить это в будущих версиях.

Используйте меню, чтобы перейти в режим редактирования. Сейчас нельзя изменить макет или значки кнопок. Я надеюсь сделать это возможным в будущей версии.

## Предустановки

Вам потребуется установить инструмент, подходящий для вашей среды рабочего стола.

Я рекомендую `ydotool`, потому что в моих тестах он показывает лучшую производительность и работает как с X11, так и с Wayland.

### ydotool

`ydotool` должен работать с любым оконным менеджером, но для него нужна фоновая служба. Если в вашем дистрибутиве есть пользовательская служба systemd, запустите её командой:

```shell
systemctl start --user ydotool
```

Чтобы служба автоматически запускалась при входе в систему, выполните:

```shell
systemctl enable --user ydotool
```

Пожалуйста, убедитесь, что вы устанавливаете достаточно новую версию `ydotool`. В версиях Ubuntu до 26.04 поставляются слишком старые версии. Обходной путь смотрите в теме обсуждения.

Обсуждение `ydotool` ведётся в этой теме: <https://github.com/stefansundin/SSHRemote/discussions/5>

### xdotool

`xdotool` предназначен для компьютеров с X11. Исторически X11 использовался на большинстве Linux-компьютеров, хотя Wayland становится всё популярнее.

Одна из особенностей X11 заключается в том, что вам может понадобиться разрешить доступ к X-серверу. Именно в этом проблема, если вы получаете ошибки вида "Authorization required". Есть несколько способов это исправить; вот два варианта, которые сработали у меня:

Если `xauth list` не показывает никаких записей, попробуйте сгенерировать файл `.Xauthority`:

```shell
xauth generate :0 . trusted
```

Если это не помогло, попробуйте выдать доступ с помощью `xhost`:

```shell
xhost +local:$USER
```

Команду `xhost` нужно будет выполнять после каждой загрузки системы. Это можно автоматизировать, создав bash-скрипт и настроив его автозапуск при входе в систему.

Обсуждение `xdotool` ведётся в этой теме: <https://github.com/stefansundin/SSHRemote/discussions/3>

### wtype

`wtype` похож на `xdotool`, но предназначен для Wayland.

Обычно он не поддерживает управление мышью, но я создал изменённую версию с поддержкой мыши. Установите её, если вам нужна поддержка мыши: <https://github.com/stefansundin/wtype>

Если вы получаете ошибку `Compositor does not support the virtual keyboard protocol`, я советую попробовать другой инструмент, например `ydotool`.

Обсуждение `wtype` ведётся в этой теме: <https://github.com/stefansundin/SSHRemote/discussions/4>

### dotool

`dotool` должен работать с любым оконным менеджером, подобно `ydotool`. В моём ограниченном тестировании он оказался значительно медленнее `ydotool`.

Обсуждение `dotool` ведётся в этой теме: <https://github.com/stefansundin/SSHRemote/discussions/6>

### cec-client

Эта предустановка является экспериментальной, так как я не смог проверить её на собственном оборудовании. Обратная связь приветствуется.

Обсуждение `cec-client` ведётся в этой теме: <https://github.com/stefansundin/SSHRemote/discussions/7>

### macOS

Есть предустановка для управления VLC на macOS с использованием команд AppleScript. Мне не известен инструмент, который поддерживал бы отправку событий клавиатуры или мыши.

Обсуждение macOS ведётся в этой теме: <https://github.com/stefansundin/SSHRemote/discussions/8>

### Android

Если на вашем устройстве Android установлен SSH-сервер, возможно, вы сможете подключиться к нему и отправлять события ввода. Это более вероятно, если на устройстве используется кастомная прошивка, например KonstaKANG на Raspberry Pi.

Мне пока не удалось добиться работы поддержки мыши.

Обсуждение Android ведётся в этой теме: <https://github.com/stefansundin/SSHRemote/discussions/9>

## «Умное» управление громкостью

При редактировании пульта в меню можно найти настройки «умного» управления громкостью. Они позволяют отображать текущую громкость компьютера в приложении и быстро устанавливать её с помощью ползунка. Вы также можете использовать аппаратные кнопки устройства, чтобы быстро отправлять команды увеличения и уменьшения громкости.

Чтение текущей громкости и установка новой громкости с помощью ползунка сейчас жёстко привязаны к использованию `pactl`.

Пакет, содержащий `pactl`, обычно называется `pulseaudio-utils` или `libpulse`.

## SSH-ключи

Вы можете импортировать или генерировать SSH-ключи в настройках приложения. Подключение с помощью SSH-ключа безопаснее, чем использование паролей.

Самый простой способ импортировать существующий SSH-ключ с компьютера — отсканировать QR-код. Для создания изображения QR-кода можно использовать программу `qrencode`. Выполните команду примерно такого вида, чтобы сгенерировать QR-код:

```shell
# Перейдите в каталог с SSH-ключами:
cd ~/.ssh

# Покажите QR-код прямо в терминале:
qrencode -r id_ed25519 -t ansiutf8

# Или создайте файл изображения:
qrencode -r id_ed25519 -o qr.png

# 4096-битные RSA-ключи слишком большие для QR-кода. С помощью gzip можно едва уместить один:
cat id_rsa_4096 | gzip | base64 | qrencode -o qr.png
```

Вы можете отправить открытые SSH-ключи на сервер с помощью функции `Отправить открытый ключ` в меню. Это добавит выбранный SSH-ключ в файл `~/.ssh/authorized_keys`. Так можно легко перейти от входа по паролю к входу с помощью SSH-ключа.

Вы можете импортировать и использовать зашифрованные SSH-ключи, но сейчас приложение не умеет их генерировать.

## Безопасность

Экспортировать или извлечь из приложения закрытую часть SSH-ключей или сохранённые пароли невозможно. Эти данные шифруются с помощью AES-256, а ключ шифрования хранится в Android Keystore. Зашифрованные данные не включаются в резервные копии Android.

В приложении нет средств отправки отчётов о сбоях. Нет телеметрии. Нет рекламы. Нет сетевых запросов, кроме SSH-подключения.

Безопасность этого приложения не проходила аудит. Если вы хорошо разбираетесь в безопасности Android или SSH, пожалуйста, посмотрите исходный код и сообщите о своих выводах в этой задаче GitHub:

<https://github.com/stefansundin/SSHRemote/issues/1>
