## Пра SSH Remote

Гэты беларускі пераклад быў зроблены GitHub Copilot (GPT-5). У ім могуць быць памылкі перакладу.

SSH Remote — гэта бясплатная праграма з адкрытым зыходным кодам, якая дазваляе дыстанцыйна кіраваць кампутарамі праз SSH.

Вы можаце цалкам наладзіць каманды, якія будуць выконвацца, і праграма змяшчае прадустаноўкі для распаўсюджаных сцэнарыяў.

Я выкарыстоўваю гэтую праграму для кіравання сваёй HTPC-сістэмай на Raspberry Pi OS. Кіраванне HTPC — асноўны сцэнар, пад які аптымізавана праграма.

Гэта не эмулятар тэрмінала, але ў крайнім выпадку яна дазволіць вам выканаць `apt-get install`.

## Пачатак працы

Калі вы хочаце выкарыстоўваць SSH-ключ для падключэння, спачатку адкрыйце налады праграмы і імпартуйце або згенеруйце ключ.

Дадайце новы хост, націснуўшы кнопку `+` унізе справа. Увядзіце звесткі для злучэння і захавайце іх.

Пры першым падключэнні вам будзе прапанавана выбраць прадустаноўку. Гэты выбар наладжвае кнопкі пульта так, каб яны зручна працавалі з рознымі тыпамі кампутараў. Апісанне даступных прадустановак глядзіце ніжэй. Калі хочаце, вы можаце пачаць без налад, выбраўшы `Без прадустаноўкі`.

Калі вы не ведаеце, ці выкарыстоўвае ваш кампутар Linux X11 або Wayland, выканайце ў тэрмінале наступную каманду:

```shell
echo $XDG_SESSION_TYPE
```

У выніку павінна вывесціся `x11` або `wayland`. Гэтую каманду трэба запускаць унутры графічнага асяроддзя.

## Пульт

Пасля падключэння вы можаце карыстацца інтэрфейсам пульта, каб адпраўляць каманды. Пераключайце ўкладкі, каб атрымаць доступ да розных спосабаў уводу.

Кожнае націсканне кнопкі выконвае каманду на хосце. Для такой простай рэчы, як націск клавішы, гэта стварае вялікія накладныя выдаткі, таму вы можаце заўважыць даволі высокую затрымку ў параўнанні са звычайнай клавіятурай. Спадзяюся палепшыць гэта ў будучых версіях.

Выкарыстоўвайце меню, каб увайсці ў рэжым рэдагавання. Зараз нельга змяняць макет або значкі кнопак. Я спадзяюся дадаць такую магчымасць у будучай версіі.

## Прадустаноўкі

Вам трэба ўсталяваць інструмент, патрэбны для вашага графічнага асяроддзя.

Я рэкамендую `ydotool`, бо ў маіх тэстах ён паказаў найлепшую прадукцыйнасць і працуе як з X11, так і з Wayland.

### ydotool

`ydotool` павінен працаваць з любым аконным менеджарам, але для яго патрэбна фонавая служба. Калі ваш дыстрыбутыў пастаўляе systemd user service, запусціце яе камандай:

```shell
systemctl start --user ydotool
```

Каб служба запускалася аўтаматычна пры ўваходзе ў сістэму, выканайце:

```shell
systemctl enable --user ydotool
```

Калі ласка, пераканайцеся, што вы ўсталёўваеце дастаткова новую версію `ydotool`. Версіі Ubuntu раней за 26.04 пастаўляюць занадта старыя версіі. Абходны варыянт апісаны ў тэме абмеркавання.

Калі ласка, абмяркоўвайце `ydotool` у гэтай тэме: <https://github.com/stefansundin/SSHRemote/discussions/5>

### xdotool

`xdotool` прызначаны для кампутараў з X11. Гістарычна большасць кампутараў Linux выкарыстоўвала X11, але Wayland становіцца ўсё больш папулярным.

Адна з асаблівасцей X11 у тым, што вам можа спатрэбіцца адкрыць доступ да X server. Гэта і ёсць праблема, калі вы атрымліваеце памылкі "Authorization required". Ёсць некалькі спосабаў выправіць гэта; вось два варыянты, якія дапамаглі мне:

Калі `xauth list` не паказвае ніводнага запісу, паспрабуйце згенераваць файл `.Xauthority`:

```shell
xauth generate :0 . trusted
```

Калі гэта не дапамагло, паспрабуйце даць доступ праз `xhost`:

```shell
xhost +local:$USER
```

Каманду `xhost` трэба будзе запускаць пасля кожнай загрузкі. Гэта можна аўтаматызаваць, стварыўшы bash-скрыпт і наладзіўшы яго аўтазапуск пры ўваходзе ў сістэму.

Калі ласка, абмяркоўвайце `xdotool` у гэтай тэме: <https://github.com/stefansundin/SSHRemote/discussions/3>

### wtype

`wtype` падобны да `xdotool`, але для Wayland.

Звычайна ён не падтрымлівае кіраванне мышшу, але я стварыў мадыфікаваную версію, якая дадае такую падтрымку. Калі вам патрэбна падтрымка мышы, усталюйце яе: <https://github.com/stefansundin/wtype>

Калі вы атрымліваеце памылку `Compositor does not support the virtual keyboard protocol`, я раю паспрабаваць іншы інструмент, напрыклад `ydotool`.

Калі ласка, абмяркоўвайце `wtype` у гэтай тэме: <https://github.com/stefansundin/SSHRemote/discussions/4>

### dotool

`dotool` павінен працаваць з любым аконным менеджарам, падобна да `ydotool`. У маіх абмежаваных тэстах ён быў значна павольнейшы за `ydotool`.

Калі ласка, абмяркоўвайце `dotool` у гэтай тэме: <https://github.com/stefansundin/SSHRemote/discussions/6>

### cec-client

Гэтая прадустаноўка эксперыментальная, бо я не змог праверыць яе на ўласным абсталяванні. Водгукі вітаюцца.

Калі ласка, абмяркоўвайце `cec-client` у гэтай тэме: <https://github.com/stefansundin/SSHRemote/discussions/7>

### macOS

Існуе прадустаноўка для кіравання VLC на macOS з выкарыстаннем каманд AppleScript. Мне невядомы інструмент, які падтрымлівае адпраўку падзей клавіятуры або мышы.

Калі ласка, абмяркоўвайце macOS у гэтай тэме: <https://github.com/stefansundin/SSHRemote/discussions/8>

### Android

Калі на вашай прыладзе Android ёсць SSH-сервер, магчыма, вы зможаце падключыцца да яе і адпраўляць падзеі ўводу. Гэта больш верагодна, калі ваша прылада выкарыстоўвае кастамную прашыўку, напрыклад KonstaKANG на Raspberry Pi.

Я пакуль не высветліў, як прымусіць працаваць падтрымку мышы.

Калі ласка, абмяркоўвайце Android у гэтай тэме: <https://github.com/stefansundin/SSHRemote/discussions/9>

## Разумнае кіраванне гучнасцю

Падчас рэдагавання пульта ў меню можна знайсці "разумныя" налады гучнасці. Яны могуць паказваць бягучую гучнасць кампутара ў праграме і дазваляць хутка змяняць яе з дапамогай паўзунка. Вы таксама можаце выкарыстоўваць апаратныя кнопкі сваёй прылады, каб хутка адпраўляць каманды павелічэння або змяншэння гучнасці.

Чытанне бягучай гучнасці і ўстаноўка новага значэння праз паўзунок цяпер жорстка прывязаныя да выкарыстання `pactl`.

Пакет, які змяшчае `pactl`, звычайна называецца `pulseaudio-utils` або `libpulse`.

## SSH-ключы

Вы можаце імпартаваць або згенераваць SSH-ключы ў наладах праграмы. Падключэнне з SSH-ключом больш бяспечнае, чым выкарыстанне пароляў.

Самы просты спосаб імпартаваць існы SSH-ключ з кампутара — адсканаваць QR-код. Для стварэння выявы QR-кода можна выкарыстаць праграму `qrencode`. Напрыклад, вы можаце выканаць такую каманду:

```shell
# Перайдзіце ў каталог з вашымі SSH-ключамі:
cd ~/.ssh

# Пакажыце QR-код у тэрмінале:
qrencode -r id_ed25519 -t ansiutf8

# Або стварыце файл выявы:
qrencode -r id_ed25519 -o qr.png

# 4096-бітныя RSA-ключы занадта вялікія для QR-кода. З дапамогай gzip можна ледзьве змясціць адзін:
cat id_rsa_4096 | gzip | base64 | qrencode -o qr.png
```

Вы можаце адпраўляць публічныя SSH-ключы на сервер з дапамогай функцыі `Адправіць публічны ключ` у меню. Гэта дадасць выбраны SSH-ключ у файл `~/.ssh/authorized_keys`. Так можна лёгка перайсці ад уваходу па паролі да ўваходу па SSH-ключы.

Вы можаце імпартаваць і выкарыстоўваць шыфраваныя SSH-ключы, але пакуль не можаце генераваць іх у самой праграме.

## Бяспека

Немагчыма экспартаваць або выняць з праграмы прыватную частку SSH-ключоў або захаваныя паролі. Гэтыя даныя шыфруюцца з выкарыстаннем 256-бітнага AES, а ключ шыфравання захоўваецца ў Android Keystore. Зашыфраваныя даныя выключаюцца з Android-рэзервовых копій.

У гэтай праграме няма сістэмы справаздач пра збоі. Няма тэлеметрыі. Няма рэкламы. Няма сеткавых запытаў, акрамя SSH-злучэння.

Бяспека гэтай праграмы не праходзіла аўдыт. Калі вы добра разбіраецеся ў бяспецы Android або SSH, калі ласка, паглядзіце зыходны код і паведаміце пра свае высновы ў гэтай GitHub issue:

<https://github.com/stefansundin/SSHRemote/issues/1>
